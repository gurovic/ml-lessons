# Workflow: issue → ветка → PR

## Когда создавать issue

| Ситуация | Issue |
|----------|-------|
| Задача ~30 мин и больше (агент, пайплайн, пересборка уроков, новый урок) | **Да** |
| Архитектурное решение, меняется несколько модулей | **Да** |
| Опечатка, однострочный багфикс | **Нет** |
| Быстрая правка в чате без последствий для репозитория | **Нет** |

Issue — **спецификация для агентов и людей**: цель, чеклист, критерии готовности, связанные файлы. Шаблоны: `.github/ISSUE_TEMPLATE/` (`task`, `lesson`).

## Порядок работы

1. **Issue** — заполнить шаблон (цель, план, DoD, файлы).
2. **Ветка** — `issue-<N>/<short-name>` от `main`, например `issue-12/link-checker-fix`.
3. **Работа** — коммиты в ветке; при необходимости обновлять чеклист в issue.
4. **PR** — в `main`, в описании строка `Closes #N` (или `Fixes #N`), чтобы issue закрылся при merge.

## Как используют issue агенты

- Читать issue **до** правок кода — это источник «что» и «готово ли».
- Не дублировать длинное ТЗ в чате, если оно уже в issue.
- В отчёте после задачи — ссылка на issue и PR.
- Для уроков: содержание слайдов и порядок — в **`plan.md`**, issue фиксирует **объём работ** (какой урок, какие шаги пайплайна).

## Issue vs plan.md

| | GitHub Issue | `lessons/<name>/plan.md` |
|---|--------------|---------------------------|
| **Область** | Задачи репозитория, пайплайн, агенты | Содержание конкретного урока |
| **Кто пишет** | Автор / ассистент при постановке задачи | Автор урока |
| **Жизненный цикл** | Закрывается PR (`Closes #N`) | Живёт с уроком, обновляется при переработке |
| **Шаблон** | `.github/ISSUE_TEMPLATE/lesson.md` | Структура слайдов в папке урока |

Issue типа `lesson` **ссылается** на `plan.md`, но не заменяет его.

## Связанные документы

- **RULES.md** — правила для ассистента, в т.ч. issue-first
- **docs/pipeline.md** — порядок шагов над уроком после `plan.md`
