# Мини-проект: project.ipynb

## Проблема

`code.ipynb` даёт короткие фрагменты кода по отдельным слайдам. На занятии не хватает **сквозного сценария**: один датасет, одна задача, все шаги ML-пайплайна от EDA до метрик и диагностики.

Отдельная проблема: `project.ipynb` не должен быть **набором изолированных демо техник**. Если на EDA увидели выбросы — следующий шаг их удаляет или обрабатывает; если сравнили Ridge и Lasso — выбирают победителя; **все последующие ячейки** работают с уже очищенными данными и выбранной моделью.

## Решение

Дополнительный файл **`project.ipynb`** в папке урока — end-to-end мини-проект на **реальных данных** (sklearn / OpenML, без внешних CSV).

### Отличие от code.ipynb

| | `code.ipynb` | `project.ipynb` |
|---|-------------|-----------------|
| Структура | по заголовкам слайдов | по этапам одного проекта |
| Данные | синтетика, мини-выборки | один реальный датасет на весь ноутбук |
| Связность | каждый блок сам по себе | **непрерывная история**, состояние передаётся между ячейками |
| Цель | показать API и идею слайда | повторить максимум тем урока в одном сценарии |

## Паттерн narrative pipeline

Каждый `project.ipynb` читается как **одна история**, а не каталог приёмов.

### Обязательные принципы

1. **Один датасет** — загрузка один раз в начале; дальше только трансформации (`df` → `df_clean` → `X_train` / `X_test`).
2. **Состояние переносится** — переменные `df`, `df_clean`, `X_train`, `X_test`, `y_train`, `y_test`, `final_model` (или `final_pipe`) создаются один раз и переиспользуются.
3. **Решения явные** — между этапами markdown-ячейки вида «**Решение:** удаляем выбросы по IQR», «**Решение:** выбираем Ridge с `alpha=1.0`».
4. **Нет ответвлений** — синтетические мини-демо на других данных (XOR, искусственные выбросы в копии `y`) допустимы только в `code.ipynb`, не в `project.ipynb`.
5. **Один hold-out test** — `train_test_split` один раз; финальные метрики и диагностика всегда на том же `X_test`, `y_test`.
6. **Pipeline без утечки** — `StandardScaler`, imputer и т.п. только внутри `Pipeline`, fit на train.

### Рекомендуемая структура

1. **Постановка задачи** — что предсказываем, какой датасет, краткий план сюжета.
2. **Загрузка и EDA** — таблица, распределения, scatter, корреляции.
3. **Проблемы данных** — выбросы, пропуски, дисбаланс → **решение** (фильтр, impute, `class_weight`).
4. **Разделение** — `train_test_split` (стратификация для классификации) на **уже обработанных** данных.
5. **Базовая модель** — алгоритм урока «из коробки» на train.
6. **Сравнение вариантов** — Ridge/Lasso, `C`, `max_depth`, `criterion` → **выбор** победителя в переменную.
7. **Финальная модель** — обучение с выбранными гиперпараметрами; дальше только она.
8. **Метрики** — hold-out test.
9. **Диагностика** — остатки / ROC–PR / калибровка / confusion matrix.
10. **Интерпретация** — веса, odds, feature importance.
11. **Чек-лист** — краткое резюме практики.

### Сюжеты по урокам

| Урок | Датасет | Narrative arc |
|------|---------|---------------|
| Линейная регрессия | `california_housing` | EDA → выбросы в Y → IQR-очистка → split → OLS vs Ridge vs Lasso → выбор → метрики → остатки → веса |
| Логистическая регрессия | `credit-g` | EDA → дисбаланс → dummy trap → `class_weight` → подбор `C` → Pipeline+scaler → ROC/PR → порог → интерпретация |
| Дерево решений | `load_wine` | EDA → split → Gini vs entropy → sweep `max_depth` → ограничение глубины → `plot_tree` → importance → confusion matrix |
| Pandas | `titanic` (OpenML) | load → EDA → impute/FE → groupby/merge → OHE → Parquet → split → Pipeline → метрики (один `df_clean` → `df_model`) |
| Визуализация | `titanic` (OpenML) | один датасет: вопрос 1 (распределения) → 2 (связи) → 3 (corr) → 4 (категории) → 5 (dashboard) → 6 (диагностика модели) |

### Правила оформления

- `random_state=42` везде, где применимо.
- Не ссылаться на номера слайдов; темы урока — по смыслу этапов.
- Графики: подписи осей на русском, `figsize` читаемый, `%matplotlib inline`.
- Один ноутбук ≈ 45–60 мин практики после лекции (не дублировать всю теорию).
- Ячейки «**Решение:**» — markdown **перед** code, который применяет решение.

### Антипаттерны (запрещено в project.ipynb)

| Антипаттерн | Почему плохо |
|-------------|--------------|
| Отдельное демо выбросов на синтетической копии `y`, а модель на исходном `df` | разрыв сюжета |
| Два несвязанных датасета в одном проекте (Breast Cancer + Credit) | не одна история |
| Новый `train_test_split` в каждой секции | разные test, нельзя сравнивать |
| Сравнение моделей без фиксации победителя в `final_model` | студент не видит итог |
| XOR / `make_classification` в конце | изолированная техника, не про основной датасет |

## Отвергнутые альтернативы

| Альтернатива | Почему нет |
|--------------|------------|
| Расширить `code.ipynb` одним большим блоком | теряется связь со слайдами; ноутбук раздувается |
| Внешние CSV | лишние зависимости и лицензии; sklearn/OpenML достаточно |
| Изолированные демо техник в project | студент не учится собирать пайплайн; только смотрит tricks |
| Авто-генератор как у notebook_generator | сценарий проекта предметный; шаблоны в build-скриптах + ручная доработка |
