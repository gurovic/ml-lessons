### План занятия: Pandas для задач ML (Концептуальный подход)

**Слайд 1.** Философия pandas: это не Excel, а numpy с индексами.
*   Ключевая идея: pandas — это удобная надстройка над numpy, а не самостоятельная сущность.
*   Для чисел внутри столбца лежит непрерывный numpy-массив (это быстро и эффективно).
*   Для строк (тип `object`) исторически это был массив указателей на Python-объекты (это медленно и неэффективно по памяти). В современном pandas стандартом становятся PyArrow-бэкенды (например, `string[pyarrow]`), которые хранят данные колоночно.
*   Главное отличие от numpy — индексы (метки строк), которые позволяют работать с данными как со словарём, а не как с матрицей.

**Слайд 2.** Индекс — это не «номер строки», а ключ.
*   Главное заблуждение новичков: путать позицию (0, 1, 2...) и метку индекса.
*   После фильтрации, merge или сортировки индексы становятся «рваными» (не 0..N).
*   `reset_index(drop=True)` — базовая операция после любой трансформации.
*   Почему это важно в ML: sklearn игнорирует индексы, но при склейке таблиц через pandas несовпадение индексов даст неожиданные NaN.

**Слайд 3.** Булева индексация — магия numpy внутри pandas.
*   Конструкция `df[df['age'] > 30]` выглядит странно для новичков.
*   Что происходит под капотом:
    1. `df['age'] > 30` создаёт numpy-массив булевых значений `[True, False, True, ...]`.
    2. Подача этого массива в `df[...]` возвращает только строки, где `True`.
*   Это прямое наследие numpy-масок. Объяснив это, мы снимаем «мистический» налёт с синтаксиса.
*   Составные условия: `&`, `|`, `~` (а не `and`/`or`!).
*   Золотое правило скобок: из-за приоритета операторов в Python каждое условие ОБЯЗАТЕЛЬНО нужно брать в скобки: `(df['age'] > 30) & (df['city'] == 'M')`. Без скобок код упадет с ошибкой или выдаст мусор.

**Слайд 4.** Три способа обратиться к данным: `[]`, `.loc`, `.iloc`.
*   `df['col']` — получение столбца. Только по имени.
*   `.iloc[i, j]` — доступ по позиции (как в numpy). Индексы игнорируются.
*   `.loc[label, col]` — доступ по метке индекса.
*   Золотое правило: присваивание (`df['new'] = ...`) всегда через `.loc`, чтобы избежать неоднозначностей.
*   Для ML: при финальной подготовке данных перед sklearn лучше переходить к `.iloc` (позиционная индексация), так как sklearn работает только с позициями.

**Слайд 5.** Самая большая боль: SettingWithCopyWarning.
*   Суть проблемы: pandas не всегда понимает, работаете вы с оригиналом или с копией.
*   Классический пример:
    ```python
    subset = df[df['age'] > 30]
    subset['new_col'] = 1  # <- Warning!
    ```
*   Почему это плохо: изменения могут не сохраниться в `df`, либо наоборот — перезаписаться случайно.
*   Решение: всегда явно указывать намерения через `.copy()` или `.loc`:
    ```python
    subset = df[df['age'] > 30].copy()  # явно создаём копию
    # или
    df.loc[df['age'] > 30, 'new_col'] = 1  # работаем с оригиналом
    ```

**Слайд 6.** View vs Copy: архитектурная проблема pandas.
*   View — представление, разделяющее память с оригиналом.
*   Copy — независимая копия данных.
*   pandas исторически решает это «по ситуации», что приводит к неопределенности.
*   Copy-on-Write (CoW): начиная с pandas 2.0, введена новая семантика — всё ведёт себя как копия, а реальное копирование происходит только при изменении. Включается через `pd.options.mode.copy_on_write = True`.
*   Для ML-инженера: всегда делайте `.copy()` при создании новых признаков, если не уверены.

**Слайд 7.** Пропуски: NaN, None, NaT и эволюция типов.
*   pandas имеет разные «пустые» значения: `np.nan` (числовой), `None` (объектный), `pd.NaT` (временной).
*   `df.isnull().sum()` — первый шаг диагностики.
*   Эволюция пропусков (`pd.NA`): Раньше появление пропуска в целочисленном столбце (`int64`) автоматически превращало весь столбец во `float64` (так как `np.nan` — это float). В современном pandas используйте Nullable-типы (`Int64` с большой буквы, `boolean`) и значение `pd.NA`. Это позволяет хранить пропуски, не ломая целочисленные типы (что критично для ID и категорий).
*   Стратегии в ML: Удаление, заполнение медианой/модой. Критическое правило: значения для заполнения считаются только по Train, иначе утечка.

**Слайд 8.** Базовая аналитика: `value_counts()`, `nunique()`, `corr()`.
*   `value_counts()` — частоты категорий. Основа EDA для категориальных признаков. Показывает дисбаланс классов и мусорные категории.
*   `nunique()` — количество уникальных значений. Критично для понимания кардинальности: если у признака 100,000 уникальных значений из 100,000 строк — это ID, его нужно удалить.
*   `corr()` — матрица корреляций между числовыми признаками. Быстрый способ обнаружить мультиколлинеарность (корреляция > 0.9) перед обучением линейных моделей.
*   `describe(include='all')` — расширенная статистика, включающая категории (частоты, топ-значения).

**Слайд 9.** GroupBy: две операции, которые должен знать ML-инженер.
*   `agg()` — агрегация: схлопывает группы в одну строку. Используется для создания признаков «среднее по группе».
*   `transform()` — возвращает результат того же размера, что и вход. Ключевая операция для контекстных признаков:
    ```python
    df['income_deviation'] = df['income'] - df.groupby('city')['income'].transform('mean')
    ```
*   Без `transform` нельзя создать признак «отклонение клиента от среднего по городу» без потери гранулярности.

**Слайд 10.** MultiIndex: иерархические индексы после groupby.
*   Когда вы делаете `df.groupby(['city', 'category']).agg(['mean', 'std'])`, результат имеет **двухуровневый индекс** (MultiIndex) по строкам и столбцам.
*   Это частая боль: студенты не понимают, как обратиться к данным, как сбросить индекс, как склеить с исходной таблицей.
*   Решение: `reset_index()` превращает уровни индекса в обычные столбцы. Альтернатива: `as_index=False` в `groupby()`.
*   Для ML: всегда сбрасывайте MultiIndex перед подачей в модель или перед merge.

**Слайд 11.** Категории: типы кодирования и когда что применять.
*   Ordinal (Label Encoding): только для упорядоченных (низкий/средний/высокий). В pandas через `astype('category')` с `set_categories()` и `.cat.codes`.
*   One-Hot: для неупорядоченных. `pd.get_dummies(df, columns=['city'], drop_first=True)`.
*   Target Encoding: замена на среднее таргета. Опасно утечкой, делать только через sklearn `Pipeline`.
*   Частый провал студентов: кодирование категорий числами 0, 1, 2... для деревьев работает, для линейных моделей — ломает геометрию.

**Слайд 12.** Быстрый Feature Engineering: условия, биннинг и отсечение.
*   Векторизованные условия (`np.where`): Мост между numpy и pandas. Работает в сотни раз быстрее `apply(lambda row: ...)`.
    *   Пример: `df['is_premium'] = np.where(df['spend'] > 1000, 1, 0)`
*   Биннинг (`pd.cut` и `pd.qcut`): Превращение чисел в категории. `cut` делает равные интервалы, `qcut` — равное количество объектов в корзинах (квантили). Помогает линейным моделям стать устойчивее к выбросам, а деревьям — проще строить разбиения.
*   Инженерная борьба с выбросами (`clip`): Вместо удаления строк с аномалиями (и потери информации по другим признакам) используйте Winsorization (отсечение).
    *   Пример: `df['income'] = df['income'].clip(lower=0, upper=df['income'].quantile(0.99))`. Мы «сплющиваем» аномальный доход до верхней границы нормы, сохраняя объем выборки и спасая модели от взрыва градиентов.

**Слайд 13.** Работа с текстом: стринговый аксессор `.str`.
*   pandas имеет мощный аксессор `.str` для векторизованных операций над строками. Это основа предварительной обработки текста до NLP.
*   Ключевые методы:
    *   `.str.len()` — длина строки.
    *   `.str.lower()`, `.str.upper()` — приведение к регистру.
    *   `.str.contains('pattern')` — поиск подстроки (возвращает булев массив).
    *   `.str.replace('old', 'new')` — замена.
    *   `.str.extract(r'regex')` — извлечение по регулярному выражению (возвращает DataFrame!).
    *   `.str.split().str[0]` — разбиение и взятие элемента.
*   Важно: `.str` работает только с типами `object` и `string[pyarrow]`.

**Слайд 14.** Время как признак: аксессор `.dt`.
*   Модель не понимает unix-timestamp или строку `2023-10-25 14:30:00`. Для неё это просто большое число.
*   Модель понимает цикличность и события. Используем `.dt` для создания смысловых признаков:
    *   `df['date'].dt.dayofweek` (чтобы ловить выходные).
    *   `df['date'].dt.hour` (утренний/вечерний спрос).
    *   `df['date'].dt.is_month_end` (флаг конца месяца, когда люди получают зарплату).
*   Важный нюанс: Циклические признаки (час, месяц) часто нужно кодировать через `sin` и `cos`, чтобы 23:00 и 00:00 были математически близки (мостик к продвинутому Feature Engineering).

**Слайд 15.** pandas <-> numpy: мост между экосистемами.
*   В numpy из pandas: `df.to_numpy()` — современный способ. `df.select_dtypes(include=np.number).to_numpy()` — только числа для моделей.
*   Из numpy в pandas: `pd.DataFrame(arr, columns=['a', 'b'])` — можно задать имена столбцов.
*   Подводный камень: если в DataFrame есть строки, `.to_numpy()` приведёт всё к `object`, и модель сломается.

**Слайд 16.** Опасности склейки таблиц: merge и concat.
*   Скрытое раздувание (Cartesian Product) в `merge`: Если при `pd.merge(A, B, on='id')` в таблице B ключи не уникальны, строки из A размножатся. Объем данных взорвется, а статистики (например, `sum`) станут неверными. Всегда проверяйте уникальность ключей перед джойном!
*   Ловушка индексов в `concat`: `pd.concat([df1, df2], axis=1)` склеивает таблицы по индексу, а не по порядку строк. Если после `train_test_split` индексы стали «рваными`, `concat` создаст кучу NaN. Всегда делайте `reset_index(drop=True)` перед склейкой по столбцам.

**Слайд 17.** merge_asof: асинхронный join для временных рядов.
*   Классический `merge` требует точного совпадения ключей. Но в реальных данных (IoT, финансы) события происходят в разное время.
*   `pd.merge_asof()` — join по ближайшему времени. Например, склеить транзакции с ближайшим курсом валют на момент транзакции.
*   Пример: `pd.merge_asof(transactions, rates, on='timestamp', direction='backward')`.
*   Критично для ML: без этого невозможно создать признаки «курс на момент покупки», «погода на момент заказа».

**Слайд 18.** pandas <-> sklearn: где возникают проблемы.
*   sklearn (в отличие от pandas) игнорирует имена столбцов и индексы. Работает с позициями.
*   Проблема 1: разный набор категорий в Train и Test -> `OneHotEncoder` даст разное число столбцов -> ошибка при `predict`. Решение: `handle_unknown='ignore'` внутри `Pipeline`.
*   Проблема 2: индексы после `train_test_split` остаются старые, и при случайной склейке могут возникнуть неожиданные NaN. Решение: `reset_index(drop=True)` после сплита.

**Слайд 19.** pandas <-> PyTorch: создание Dataset.
*   Стандартный путь: pandas -> numpy -> `torch.from_numpy()` -> `TensorDataset`.
*   Важный нюанс памяти: `torch.from_numpy(X)` не копирует данные, а создает тензор, разделяющий память с numpy-массивом. Случайное изменение numpy-массива изменит и тензор. Если нужна независимость данных, используйте `torch.tensor(X)` (он создает копию) или `.copy()`.
*   Важно: явно задавать `dtype`, иначе PyTorch будет ругаться на object/int64 для float-тензоров.

**Слайд 20.** Data Leakage: где прячется в pandas.
*   Утечка при импутации: `df['col'].fillna(df['col'].mean())` до сплита — классическая ошибка.
*   Утечка через Target Encoding: использование таргета для создания признака вне Pipeline.
*   Утечка через индексы:
    1. Индекс сам несет сигнал (например, это `timestamp` или `user_id`), и модель неявно использует его, если мы забыли сделать `reset_index()`.
    2. Случайный `join` по старым индексам после сплита при неаккуратной склейке таблиц.
*   Решение: все трансформации, зависящие от данных, делать внутри `sklearn.Pipeline`. pandas — только для первичной загрузки и EDA.

**Слайд 21.** Итеративная обработка: когда файл не влезает в RAM.
*   Если CSV весит 50 ГБ, а RAM 16 ГБ, `read_csv()` упадет с OOM.
*   Решение: параметр `chunksize` в `read_csv()`. Он возвращает итератор, который читает файл порциями.
    ```python
    for chunk in pd.read_csv('huge.csv', chunksize=100000):
        # обрабатываем chunk, накапливаем статистики
        process(chunk)
    ```
*   Это базовый навык для работы с большими данными без Dask/Spark.
*   Альтернатива: Parquet с фильтрацией через `pyarrow` (читает только нужные столбцы/строки).

**Слайд 22.** Профилирование данных: автоматический EDA.
*   Ручной EDA через `info()`, `describe()`, `value_counts()` занимает часы.
*   Библиотека `ydata-profiling` (бывшая `pandas-profiling`) генерирует HTML-отчет одной командой:
    ```python
    from ydata_profiling import ProfileReport
    profile = ProfileReport(df, explorative=True)
    profile.to_file("report.html")
    ```
*   Отчет включает: распределения, корреляции, пропуски, дубликаты, предупреждения о проблемах.
*   Это стандарт индустрии для быстрого понимания нового датасета.

**Слайд 23.** Производительность: когда pandas «не тянет».
*   pandas — однопоточный и не оптимизирован для больших данных.
*   Признаки деградации:
    *   Итерация по строкам (`iterrows`, `itertuples`) — всегда заменяйте на векторизацию.
    *   `apply` с Python-функцией — в 100 раз медленнее векторизованной операции.
*   Первая линия обороны: векторизация. Вместо `df['len'] = df['text'].apply(len)` писать `df['len'] = df['text'].str.len()`.

**Слайд 24.** Оптимизация памяти: downcasting типов.
*   По умолчанию pandas использует `float64`, `int64` — это расточительно.
*   Приёмы:
    *   Категории вместо `object`: `df['city'] = df['city'].astype('category')` экономит в 10-100 раз.
    *   `float32` вместо `float64` — достаточно для ML.
    *   Автоматический downcasting через `pd.to_numeric(df['col'], downcast='integer')`.

**Слайд 25.** Форматы хранения: Смерть CSV и восстание Parquet.
*   CSV не сохраняет типы данных. Целочисленный ID `007` превратится в `7`. Дата превратится в строку. Категории станут объектами. При повторной загрузке вся работа по оптимизации типов будет уничтожена.
*   CSV читается и пишется экстремально медленно на больших датасетах.
*   Золотой стандарт ML-инженера: Parquet (колоночный формат, тесно связанный с PyArrow). Он сохраняет типы pandas (включая `category` и `datetime`), сжимает данные в 5-10 раз и читается мгновенно.
*   Правило: CSV — только для отправки отчета бизнесу или первичного получения сырых данных. Вся внутренняя кухня ML-пайплайна (сохранение после очистки, передача между этапами) — только через `.to_parquet()`.

**Слайд 26.** Экспорт в продакшен: SQL и другие форматы.
*   `to_sql()` — запись в базу данных через SQLAlchemy. Стандарт для интеграции с продакшен-системами.
    ```python
    df.to_sql('predictions', engine, if_exists='append', index=False)
    ```
*   `to_json()` — для REST API и веб-интеграций.
*   `to_feather()` — быстрый бинарный формат (альтернатива Parquet для промежуточных данных).
*   Важно: всегда указывайте `index=False` при экспорте, иначе индекс станет лишним столбцом.

**Слайд 27.** Альтернативы: что использовать, когда pandas не справляется.
*   Polars: написан на Rust, ленивые вычисления, в 10-100 раз быстрее pandas. Похожий API.
*   Dask: «распределённый pandas», для данных больше RAM.
*   DuckDB: SQL-движок для аналитики поверх файлов.
*   В индустрии pandas остаётся стандартом для EDA и небольших датасетов (< 1-2 ГБ), для продакшена всё чаще Polars/DuckDB.

**Слайд 28.** Pandas и GPU: разрушение мифов и экосистема RAPIDS.
*   Главный миф: «pandas можно ускорить, просто подключив GPU». Реальность: нативный pandas работает строго на CPU и не умеет обращаться к видеокарте.
*   Экосистемное решение: библиотека cuDF из проекта NVIDIA RAPIDS. Она предоставляет API, почти идентичный pandas, но выполняет операции (`merge`, `groupby`) на GPU, что дает ускорение в десятки раз на больших данных.
*   Современный подход: модуль `cudf.pandas`, который автоматически перехватывает вызовы и отправляет их на GPU, если это возможно, иначе использует CPU.
*   Связь с Deep Learning: pandas готовит данные в оперативной памяти (RAM). Чтобы нейросеть обучалась на GPU, данные нужно явно передать в видеопамять (VRAM) через PyTorch (`tensor.to('cuda')`). Сам pandas в VRAM не работает.

**Слайд 29.** Гигиена памяти в Jupyter: борьба с OOM (Out Of Memory).
*   Студенты работают в Jupyter Notebook. Они делают `df1 = ...`, потом `df2 = df1...`, потом переопределяют `df1`.
*   Jupyter хранит в скрытых переменных (например, `_`, `__` или в истории выводов) ссылки на старые огромные датафреймы. В итоге RAM забивается, и ядро падает с Out of Memory на этапе обучения модели.
*   Решение: Явное управление памятью. Использование `del df_old` для удаления ненужных промежуточных таблиц и вызов `import gc; gc.collect()` для принудительного запуска сборщика мусора Python перед запуском тяжелой модели.

**Слайд 30.** Итоговая ментальная модель pandas для ML-инженера.
*   pandas = инструмент EDA и первичной очистки. Не используйте его для тяжелых трансформаций в продакшене.
*   Индекс — это ключ. Всегда помните о нём при join/merge/concat.
*   View/Copy — всегда `.copy()` или `.loc`. Избегайте `SettingWithCopyWarning`.
*   Векторизация > apply > iterrows. В этом порядке падает производительность.
*   Все трансформации, зависящие от данных — в sklearn Pipeline. pandas только загружает и показывает.
*   Перед моделью — `.to_numpy()` и `reset_index()`. Переводим в «чистый» формат.
*   Сохраняем промежуточные данные только в Parquet.
*   pandas работает только в RAM. Для вычислений на GPU данные нужно передавать в специализированные библиотеки (cuDF, PyTorch).
